# ACCA Building And Development Process Write-Up

Date: 2026-05-12

This write-up explains what ACCA/D-ACCA is, how it was built, where the work can go next, how it may become a startup or SaaS wedge, and how it can connect with adjacent IVY projects such as Signal and Recall Board. The section sizes follow the requested shape: roughly 750 words on what it is, 750 on the build process, 750 on future prospects, 500 on startup/SaaS basis, and 250 on integrations.

## 1. What ACCA Actually Is

ACCA is best understood as an auditable, authority-constrained context compiler for AI agents. It is not a search engine in the normal sense, and it is not a replacement for a large language model. Its job is narrower and more important: given a user turn, an agent turn, a project state, a pile of memory, and a set of possible evidence records, it decides what context should be admitted into the model's prompt, what should be rejected, what should be treated as stale or dangerous, and when the correct answer is no context at all. The core product idea is not "find text similar to the query." It is "govern what an AI is allowed to remember, trust, and use right now."

That distinction is the reason D-ACCA keeps beating naive BM25 on the important metrics. BM25 is a retrieval method. It scores documents by lexical overlap. It is fast and useful as a candidate generator, but it has no built-in concept of authority, freshness, contradiction, user intent, safety exposure, source hierarchy, or answerability. If the query mentions "latest price," BM25 may happily return an old price if the old document has more matching words. If a decoy record repeats the right keywords, BM25 may select it. If a memory says something that used to be true, BM25 has no reason to reject it. D-ACCA exists because agent context is not only a recall problem. It is an admission problem.

The current engine has several conceptual layers. The first layer is candidate retrieval: possible evidence records are found from a corpus of docs, memory records, runbooks, generated notes, and project artifacts. The second layer is deterministic routing: rules and scores decide which candidates are plausibly relevant. The third layer is authority and freshness gating: current source files, active docs, recent run outputs, and explicit user instructions should outrank stale memory, generated summaries, and decoys. The fourth layer is safety and exposure control: private, tainted, secret-like, or forbidden records should not leak into the frontier context packet. The fifth layer is compact packet assembly: the model should receive the smallest useful evidence set, not a pile of loosely related text. The sixth layer is route proof generation: the system records what was selected, rejected, overflowed, or abstained from, and why.

D-ACCA is the deterministic form of this engine. It tries to make the hot path fast, auditable, and predictable. DD-ACCA and the librarian variants add a second idea: sometimes deterministic routing needs help interpreting messy human or agent language. A user may say "the phone kitty litter thing is stuck," and the right memory may live under a Signal/Tailscale/KittyLitter setup note. A deterministic engine can handle that if it has learned aliases, but it may miss the intent if no alias exists. The librarian's job is to translate messy phrasing into better candidate queries, identify ambiguity, and warn about stale/current conflicts. Crucially, the librarian does not get to admit evidence directly. It proposes. D-ACCA verifies. This is what keeps the system from becoming another opaque model call.

The recent helper-lazy and Spec-DD-lazy work clarified the architecture. The fastest version looks like speculative inference applied to context routing. A cheap deterministic draft proposes a query or context family, and the D-ACCA verifier decides whether the resulting evidence is admissible. Most easy cases should finish without waking a model. Hard cases can escalate to a sub-frontier librarian such as DeepSeek Flash, but only in shadow, parallel, or rare blocking modes. DeepSeek is valuable as a teacher and failure explainer; it is too slow to sit on every request.

This makes ACCA more like an engine than a benchmark harness. Its input is a real agent/user moment. Its output is an evidence decision: accepted packet, no-context decision, abstention, or escalation. It is useful because modern agents fail when they use the wrong context, not only when they lack context. Long-context windows make this problem worse in some cases because they invite contamination. ACCA's claim is that smaller, better-governed packets can outperform larger, dirtier context.

So the short definition is: ACCA is a low-latency context and memory admission engine for AI agents. It selects, rejects, compacts, and explains evidence before the frontier model answers. D-ACCA is the deterministic fast path. DD-ACCA and librarian variants are adaptive sidecars used to recover intent and improve hard cases without giving up deterministic final admission.

## 2. The Building Process

The development process started from a practical frustration: agent memory and retrieval are too easy to make impressive on internal examples and too easy to break in real work. Early checkpoints built the basic shape of a context stress lab. Instead of asking whether the router could retrieve something relevant, the tests asked whether it could retrieve the right thing while avoiding stale, decoy, or forbidden evidence. That shifted the benchmark from search quality toward context governance quality.

The first major build phase established the schema and artifact discipline. Corpus items needed ids, source families, tags, timestamps, provenance, and exposure metadata. Eval cases needed required evidence, forbidden evidence, expected answerability, and categories such as stale conflict, no-context, safety, or authority hierarchy. The runner needed to emit route proofs and frontier packets rather than only a pass/fail number. This mattered because ACCA's long-term value depends on auditability. A normal retriever can say "here are the top five hits." ACCA needs to say "these were selected, these were rejected, this stale record was known but not admitted, and this packet is safe to show the model."

The second phase compared D-ACCA against baselines. Naive BM25, source-family BM25, exact-anchor baselines, and long-context-style retrieval were used as pressure tests. These baselines were important because they are plausible simple alternatives. If a cheap lexical retriever plus a bigger context window solved the problem, D-ACCA would not be worth building. The results moved in the opposite direction. BM25 often had high recall but terrible contamination: many stale extras, decoy extras, and forbidden hits. D-ACCA's strength was not retrieving more. Its strength was admitting less and admitting better.

The third phase pushed scale and latency. Python indexed routing could prove behavior but was not the final latency story. Rust backend probes and staged pilots showed that the same selected-evidence behavior could be preserved with much lower warm-route latency. The important milestone was not just a single fast run; repeated runs showed stable behavior in the low-millisecond band. That supported the thesis that a context admission engine can be fast enough to sit in front of interactive agents.

The fourth phase moved from internal synthetic stress to more IVY-real and model-facing cases. The generated IVY-real datasets pulled from actual repo concepts: hot-session cache reuse, MoME opt-in memory, llama.cpp paths, sandbox policies, Qwen model tracks, and command runbooks. This made the tests more natural, but still controlled. Model-facing demos then produced no-memory, naive BM25, and ACCA packet prompts so the work could eventually measure final answer quality, not just retrieval metrics.

The fifth phase focused on hard failures. Reports and staged tests showed recurring weakness around "latest/current" claims, unsupported external facts, no-anchor queries, and ambiguity. These failures were valuable because they were not random. They pointed to missing answerability gating and better intent recovery. That led to the librarian idea: a sub-frontier model or deterministic sidecar could act like a librarian, not as the final authority, but as a query compiler and ambiguity scout.

The sixth phase built and tested librarian variants. The fixture librarian proved the harness. The rule librarian provided a deterministic baseline. The live DeepSeek Flash librarian proved that a model could translate vague phrasing into useful canonical query bundles, but its roughly tens-of-seconds latency made it unsuitable for the hot path. DD-rule distilled the useful DeepSeek behavior into deterministic rules. Spec-DD applied speculative-decoding and multi-token-prediction style thinking: draft several likely query heads, verify them through D-ACCA, and accept only evidence-bearing heads. Spec-DD-lazy improved the latency shape by deferring verification to the final D-ACCA bundle route.

The seventh phase built helper-lazy and the black-box packet tests. The 1000-case mixed packet and 700-case edge-only packet tested the variants against user/agent input without caring what engine produced the answer. The packet included normal cases, wacky phrasing, stale traps, decoys, no-context prompts, aliases, and current-only requests. Helper-lazy reached perfect scores on those packets because it used learned alias/helper metadata and still left final admission to D-ACCA. That was a useful architecture proof but not an external generalization proof. It showed that a learned profile/librarian layer can be powerful if the memory profile contains good routing hints.

The current build process has therefore produced more than code. It has produced an experimental method: build a routing feature, compare it against naive retrieval, force stale and forbidden traps, measure latency, persist route proofs, document failure modes, and only then decide whether to promote a variant. That discipline is the real asset.

## 3. Future Prospects And Leads

The next stage should be less about making synthetic scores prettier and more about making the engine survive natural developer work. The current results show that D-ACCA has a strong internal shape, but they also show where overfitting can creep in. Helper-lazy reaching 1.0 on the black-box packet is encouraging, but the packet was generated by our own system and included metadata that helper-lazy could exploit. The next proof has to remove some of that support and ask whether the engine still behaves well.

The first future lead is metadata ablation. Remove aliases, helper queries, or guard terms from some corpus records and measure how far helper-lazy falls. This answers whether the system is truly understanding enough routing structure or simply reading curated metadata. A good result would not be "no drop." A realistic result would show graceful degradation: direct D-ACCA handles exact and current cases, helper-lazy handles known aliases, and the librarian wakes up for missing-alias cases.

The second lead is held-out alias testing. The system should see records about Signal, Recall Board, hot-session cache, or MoME memory, but the test queries should use phrases never included in the metadata. For example, "phone wakeup thing," "diagram brain," or "context stuffing poison" should be mapped to the right memory family only if the engine has enough semantic or learned structure. This is where a sub-frontier librarian may justify itself. If DeepSeek can solve those cases slowly, and the distillation loop can convert the solutions into deterministic aliases, the engine begins to learn.

The third lead is real conversation replay. Codex session JSONL files are available locally, and OpenCode state appears at least partially discoverable through app storage. These logs can be mined into real user/agent turns, tool actions, follow-ups, ambiguous "do it" commands, and situations where the agent had to remember previous work. The key is to avoid committing raw private logs. The harness should extract derived cases, redact secrets, and store only task text, expected context family, evidence ids, and labels. A small review packet of ambiguous cases can be saved for human labeling later.

The fourth lead is Confidence Gate v1. The engine needs a deterministic decision layer that returns accept, no_context, or escalate_librarian. Candidate features include top score, score margin, alias strength, stale/current conflict count, forbidden hit risk, source authority, query-to-evidence lexical gap, no-context pattern match, and whether the selected evidence actually supports answerability. This gate is more important than the model. A good librarian called too often becomes a latency tax. A good librarian called too rarely fails to help. The confidence gate determines the whole runtime economics.

The fifth lead is contrastive evidence. The engine should not only ask what matches; it should ask what proves the selected answer over nearby wrong answers. Current vs stale, allowed vs forbidden, exact entity vs near entity, and project-specific answer vs generic answer should be compared directly. This will help with the known CP26/CP27 class of failure: unsupported latest/current claims that should abstain instead of selecting plausible nearby context.

The sixth lead is a distillation log. Every time the librarian helps, the system should write a durable candidate lesson: new alias, negative constraint, freshness rule, contradiction pattern, held-out test case, or confidence-threshold adjustment. The goal is not to make DeepSeek part of every request. The goal is to let adaptive intelligence solve hard cases and then compress repeated lessons into the deterministic engine. This is where D-ACCA starts looking like a memory engine rather than a static router.

The seventh lead is context packet pruning and answer-side evaluation. Retrieval metrics are necessary but not enough. The model's final answer should be tested under no-context, BM25, D-ACCA, and librarian-assisted packets. Did the packet improve the final patch, reduce tool calls, avoid stale claims, or help the agent cite evidence? A context engine that improves routing but does not improve answers is academically interesting but product-weak. The next serious harness should measure final agent outcomes.

The eighth lead is shadow field mode. D-ACCA should run passively beside Codex/OpenCode sessions and record what it would have injected, without changing the workflow. Later, compare proposed packets against what the agent or developer manually searched for. This is the most honest product test because it measures whether the engine predicts real context needs before the human or agent has to hunt for them.

The long-term prospect is a continuously learning context governance engine. The fast path stays deterministic. The slow path handles ambiguity. The learning path distills slow wins into fast rules. That architecture can support personal memory, team memory, coding-agent context, and eventually multi-agent context exchange.

## 4. Startup And SaaS Basis

The startup wedge is not "another AI search tool" and not "another coding assistant." The wedge is context governance for AI agents. Teams already have agents, copilots, IDE assistants, chat logs, docs, repositories, issue trackers, and internal runbooks. The pain is that these tools often use the wrong context, miss current context, over-trust stale notes, or cannot explain why a piece of evidence was used. D-ACCA can sit beside existing tools and make context selection safer, smaller, and more auditable.

The best first product is probably a shadow recorder and context auditor. Instead of asking users to replace Codex, OpenCode, Cursor-like tools, or internal agents, install D-ACCA as a passive sidecar. It watches allowed local/project signals, routes candidate context, produces route proofs, and reports what it would have injected. That avoids the adoption risk of changing the agent's behavior on day one. It also produces the most valuable dataset: real tasks, real missed context, real stale traps, and real proof artifacts.

The second product tier is assisted injection. The user or team can preview a D-ACCA packet and choose to inject it into an agent. This is safer than automatic injection because it builds trust and creates accept/reject labels. Once accepted packets strongly outnumber rejected packets, automatic injection can be enabled for low-risk tasks. High-risk tasks can still require preview or blocking librarian verification.

The third tier is team memory governance. A company could index repositories, docs, runbooks, decision records, CI logs, release notes, and approved chat summaries. D-ACCA would enforce authority hierarchy: current repo state over stale docs, policy over memory, raw logs over generated summaries, tenant boundary over everything. The product could expose route proofs, stale evidence reports, memory lifecycle controls, and context-quality dashboards. This is where the SaaS value becomes clearer because team context failures are expensive.

Pricing could be per developer seat, per indexed repository, per routed task, or by agent-session volume. A practical path would start with a local/private-cloud developer preview, then move to hosted dashboards and team administration. Security posture matters. Raw private logs should not be uploaded casually. The product should support local-first or private deployment for sensitive teams, with SaaS used for dashboards, policy management, and aggregated non-sensitive metrics when allowed.

The strongest buyer story is reliability and trust, not speed alone. The metrics that matter are fewer stale-context mistakes, fewer unsupported claims, lower token spend, better route explanations, fewer manual searches, and better agent task outcomes. Speed helps because the system can be inserted into real workflows, but the business value is accuracy under context pressure.

The main risk is that the product becomes a fancy benchmark without proving final answer improvement. That is why the next tests should include agent A/B tasks and shadow mode. If D-ACCA can show that existing agents produce better patches, fewer wrong assumptions, and more faithful answers when given D-ACCA packets, it becomes infrastructure. If it only wins internal routing metrics, it remains a lab tool. The product path depends on crossing that line.

## 5. Integrations With Signal, Recall Board, And Adjacent Projects

Signal fits D-ACCA as the human escalation and notification layer. The local Signal project already provides a private push/reply protocol using a local daemon, Tailscale Serve, Web Push, and a phone PWA. D-ACCA can use this when confidence is low, when a memory write is risky, or when the engine needs a human label. For example: "This case looks stale/current ambiguous. Should I admit the old runbook, abstain, or ask the librarian?" Signal turns context governance into an interruptible human-in-the-loop workflow without requiring a public cloud service.

Recall Board fits as the visual memory and second-brain surface. It already exports Excalidraw boards into compact AI-readable graph/context JSON rather than screenshots. D-ACCA could route board nodes, arrows, labels, transcript summaries, and project diagrams as structured evidence. In the other direction, D-ACCA route proofs could be rendered into Recall Board as visual maps: selected evidence, rejected stale evidence, contradictions, and confidence gates. That would make memory governance inspectable instead of buried in JSON.

Together, Signal and Recall Board point toward a broader system: D-ACCA as the engine, Signal as the mobile/human control plane, and Recall Board as the visual knowledge surface. Codex, OpenCode, and other agents would connect through plugins or local APIs. The model would not need unlimited raw context. It would ask the engine for governed packets, escalate through Signal when uncertain, and leave durable visual traces in Recall Board when the work creates reusable knowledge.
