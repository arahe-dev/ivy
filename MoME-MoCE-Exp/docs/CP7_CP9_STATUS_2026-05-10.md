# CP7/CP8/CP9 Status

Date: 2026-05-10

## Completed

CP7 and CP8 are implemented and verified. CP9 has a working Rust candidate-index prototype and probe. CP9.1 now integrates Rust as an optional candidate backend while keeping Python as the proof-producing authority.

### CP7: Ivy-real mini dataset

Added `scripts/generate_ivy_real_dataset.py`.

The generated dataset lives at:

- `out/context_stress_ivy_real/corpus/corpus_items.jsonl`
- `out/context_stress_ivy_real/eval/cases.json`
- `out/context_stress_ivy_real/metadata/dataset_manifest.json`

It contains 37 hand-curated IVY evidence items and 30 labeled cases across all 10 benchmark categories. The cases cover IVY identity, ACCA framing, hot-session cache rules, MoME opt-in behavior, commands, model-track notes, safety boundaries, agent-loop workflow, stale conflicts, decoys, and no-context abstentions.

Schema/validator updates:

- `schemas/dataset_manifest.schema.json` now accepts `ivy_real` scale and the Ivy-real generator.
- `schemas/corpus_item.schema.json` allows provenance paths under the real IVY repo segments used by the dataset.
- `scripts/validate_context_stress_dataset.py` recognizes `ivy_real`.

### CP8: indexed Python backend

Added `CorpusIndex` to `scripts/mome_moce_harness.py` and a stable CLI switch:

```powershell
.\.venv\Scripts\python.exe scripts\mome_moce_harness.py --dataset out\context_stress_stress --candidate-backend indexed
```

The indexed backend builds:

- token postings;
- source-family indexes;
- ID maps;
- conflict/supersession neighbor expansion.

It only replaces candidate generation. Final scoring, admissibility gates, compact selection, route proofs, frontier packets, artifact validation, and evaluation stay on the same ABI as the scan backend.

### CP9: Rust candidate-index prototype

Added:

- `rust/acca_index/Cargo.toml`
- `rust/acca_index/src/main.rs`
- `scripts/run_rust_index_probe.py`

The Rust CLI loads the corpus JSONL, builds an inverted index, scores candidates, expands conflict/supersession neighbors, and emits JSON candidates for either one query or all eval cases.

The Python probe checks required evidence recall@k against labeled cases. This is intentionally a candidate-recall probe, not a full ACCA route-proof replacement.

### CP9.1: direct and batch-preloaded Rust backend

Updated `scripts/mome_moce_harness.py` so `--candidate-backend rust` calls the compiled Rust binary directly. The Rust backend still proposes only candidate IDs; Python still owns scoring, gates, selection, route proofs, frontier packets, and artifact validation.

The first direct version shelled out once per query. That preserved quality but was too slow on stress. The current version uses Rust `--cases` batch mode to preload benchmark candidates once per dataset during `benchmark()`, then serves per-query candidates from an in-process Python cache.

Key limitation: arbitrary one-off Rust queries still use per-query process spawn. The next optimization is a persistent process or library binding.

## Result

CP7 gives a small real-IVY proof point beyond the synthetic generator. CP8 gives a faster indexed Python candidate backend without changing route proofs or context packets. CP9 proves that a Rust candidate index can preserve required-evidence recall. CP9.1 wires Rust behind the existing `candidate_backend` interface and removes per-query process/index reload for benchmark runs through batch preloading.

## Next Recommended Work

1. Improve Rust/Python raw candidate parity while preserving selected-evidence parity.
2. Replace batch Rust process calls with a persistent process or library binding for arbitrary interactive queries.
3. Expand Ivy-real v3 with actual sanitized run outputs and failure cases before claiming broader reliability.
4. Add answer-level model checks that consume only the frontier packet and cite selected evidence IDs.
